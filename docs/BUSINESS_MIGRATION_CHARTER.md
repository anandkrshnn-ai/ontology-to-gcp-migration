# Palantir to GCP: Business Migration Charter

This document is the **North Star** for the migration program. It bridges the technical execution with enterprise strategy, defining exactly *why* we are migrating, *how* the current system behaves, the strict policies governing the transition, and the phased retirement plan. 

Without this charter, the migration risks being perceived as mere "IT plumbing" rather than a strategic business transformation.

---

## 1. Existing System (The "As-Is" State)
**Owner:** Executive Comms & Enterprise Architecture

Palantir Foundry currently operates as the central nervous system for DHL's logistics and analytics. However, it functions as a **"Walled Garden"**.
- **Data Monolith:** All source systems must ingest data into Foundry to be utilized, creating massive duplication and latency.
- **Licensing Bottlenecks:** Compute and storage are inextricably linked to Palantir's proprietary licensing model, meaning M&A (acquisitions) or B2B tie-ups require extending costly Palantir licenses to external partners.
- **System Behavior:** 
  - *SLA Breach Detection* is heavily reliant on Palantir's Magritte streaming and Pipeline Builder logic.
  - *Write-backs* occur through proprietary Workshop Actions mutating the Foundry Ontology.
  - *Dashboards* are built in Slate and Contour, locking KPI reporting into the Palantir ecosystem.

## 2. Business Plan & New Requirements (The "To-Be" State)
**Owner:** Streaming & Observability Squads

The primary driver for leaving Palantir is unlocking **Ecosystem Agility**. DHL requires an open architecture that supports rapid M&A integration and supplier data sharing without imposing software prerequisites on partners.

**New Strategic Requirements:**
- **Open Standards:** Data must reside in standard formats (e.g., Iceberg/Parquet in BigQuery) queryable by any tool (Snowflake, SAP, Tableau) without movement.
- **API-First Composability:** Internal business logic (e.g., shipping delays, customs clearance) must be containerized and exposable via REST APIs (Apigee) for B2B integrations.
- **Operational SLAs:** Sub-10ms latency for operational dashboards (as proven in the Logistics SLA Pilot).
- **Predictive Agility:** The ability to natively integrate with open-source and state-of-the-art AI models (Vertex AI) rather than waiting for vendor-specific AI rollouts.

## 3. Policies & Governance
**Owner:** Governance & Infosec Squad

The migration will not diminish DHL's security posture. All regulatory frameworks currently enforced within Palantir's proprietary security markings will be mapped 1-to-1 onto GCP's native security primitives.

**Compliance Framework Mapping:**
- **GDPR (Data Privacy):** Handled via BigQuery column-level encryption and Dataplex Policy Tags (auto-generated by the metadata scraper).
- **HIPAA (Health Logistics):** For pharma/cold-chain shipments, GCP BAA-covered services (BigQuery, Cloud Run, Pub/Sub) will be utilized with strict IAM isolation.
- **TAPA (Supply Chain Security):** Transported Asset Protection Association requirements for auditing and access control are met using Cloud Audit Logs and Secret Manager token rotation (30-day enforcement).
- **ISO 27001 (Information Security):** Infrastructure-as-Code (Terraform) validation and semantic gating (`pyspark_hash_validator.py`) ensure no unauthorized system mutations.

## 4. Execution Strategy: The Strangler Fig
**Owner:** Data Engineering + CQRS Squad

We will **NOT** execute a "Big Bang" cut-over. Such approaches risk KPI drift and analyst downtime. We will use the **Strangler Fig pattern**, phasing out components gradually while running parallel validation.

1. **Phase 1: Extract Data (Underpinning)**
   - Sync raw data from Palantir to BigQuery. Analysts still use Palantir, but the data lake shifts beneath them.
2. **Phase 2: Parallelize Logic (The Trunk)**
   - Port Contour and PySpark pipelines to Dataform and Dataproc. Run semantic hash checks daily to prove mathematical parity between Palantir outputs and GCP outputs.
3. **Phase 3: Replatform Actions (The Branches)**
   - Migrate Workshop Actions to Cloud Run + Cloud SQL.
4. **Phase 4: Cut-over UI (The Leaves)**
   - Route analysts to Streamlit/React dashboards.

## 5. Retirement Plan
**Owner:** All Squads

As the Strangler Fig takes hold, Palantir components will be formally retired in the following sequence:

| Palantir Component | GCP Replacement | Retirement Milestone |
| :--- | :--- | :--- |
| **Contour / Pipeline Builder** | BigQuery / Dataform (`.sqlx`) | Phase 2 Sign-off (Parity Proven) |
| **Code Repositories (PySpark)** | Dataproc Serverless | Phase 2 Sign-off |
| **Workshop Actions (Mutations)** | Cloud Run (FastAPI/Go) + Cloud SQL | Phase 3 Sign-off |
| **Slate / Workshop UIs** | Streamlit / Custom React | Phase 4 Sign-off |
| **Foundry Ontology** | BigQuery (Reads) + Cloud SQL (Writes) | Final License Termination |
